# Product Owner Review Skill

Use this when the user wants a senior product-owner / PM perspective on a codebase,
not just a code-correctness pass. The goal is to find things that hurt users, trust,
activation, and retention — then output a prioritized, actionable list.

## When to use
- After building features ("all thing done? give suggestions").
- "Review the project", "act as a product owner", "what should we improve".
- Before a release / when preparing a roadmap.

## Process
1. Read the project end-to-end: architecture (UI layer, ViewModel, data/Room),
   entry points (MainActivity, manifests, CI workflow), and any README.
2. Trace critical user flows and data-integrity paths:
   - Does a confirmed/imported record update derived state (balances, totals) correctly?
   - Can a delete orphan or corrupt related data (categories, accounts)?
   - Is there any path that silently miscounts money?
3. Evaluate activation friction: how hard is first-run setup? Auto-detect vs manual?
4. Evaluate trust: backups/export, edge cases in date/math/localization.
5. Group every finding into:
   - **P0** — data safety / activation blockers (must fix before more users)
   - **P1** — high-value UX/features
   - **P2** — polish / docs / nice-to-have
6. Output as a concise bulleted list with file references where relevant.
7. Offer to convert the list into a tracked todo list, and recommend which to
   implement first (default: data-safety items P0-1, P0-3, plus any exposed-but-safe
   delete path).

## Output format
```
## What's solid ✅
- ...

## Prioritized suggestions
### P0 — Must fix
1. ...
### P1 — High value
...
### P2 — Polish
...

## Strategic
...
```

## Turning into a todo list
When the user says "add then in SKILL file be reusable" or "make a todo list from
your suggestions", call `todowrite` with one item per suggestion using the
`P0-1:`/`P1-4:` style prefixes so priorities are visible in the list. Keep each
item as a single actionable sentence.

## Project-specific notes (Maldar app)
This skill was authored against the **Maldar** personal-finance Android app
(Compose + Room + KSP, Farsi/RTL, Toman currency, Jalali calendar, SMS-based
transaction capture). Known architecture:
- `ui/screens/*` Composables, `ui/theme/Theme.kt` colors, `ui/navigation/NavGraph.kt`
- `viewmodel/FinanceViewModel.kt`, `data/*` (Room entities/DAOs/repo)
- `sms/SmsParser.kt` (Rial→Toman) + `sms/SmsReceiver.kt` (sideloaded SMS path)
- `sms/BankNotificationListenerService.kt` (Play-Store-compatible notification path)
- `util/AppStrings.kt` (Farsi strings), `util/Money.kt`, `util/JalaliCalendar.kt`,
  `util/UpdateChecker.kt`, `util/SmsInboxReader.kt` (recent-sender detection +
  per-account "Show SMS" reconciliation), `util/DataExport.kt` (CSV/JSON),
  `util/CrashLogger.kt`
- `ui/screens/{ExportScreen,ImportScreen,SettingsScreen,CrashLogScreen,UpdateDialogs,AccountSmsScreen,MonthTrendGraph}.kt`
- `worker/{LoanReminderWorker,DailyReminderWorker}.kt` (+ `DailyReminderScheduler`)
- CI: `.github/workflows/build-apk.yml` builds the **release** APK (R8 shrink +
  debug-keystore signed) and publishes a GitHub prerelease tagged by the
  dynamically computed `versionName`. `versionName`/`versionCode` are derived
  from conventional commits since the last `vX.Y[.Z]` tag in
  `app/build.gradle.kts` — no manual bump needed.
- Restricted `READ_SMS`/`RECEIVE_SMS` → sideloaded APK; `BankNotificationListenerService`
  is the Play-Store-compatible alternative (declared in the manifest).

Reusable lessons:
- Hand-rolled date converters (Jalali) need unit tests around leap years/boundaries.
- Comparing release versions should use `versionCode` (Int), not parsed `versionName`.
- Parser-derived amounts must convert to the app's stored unit (Rial/10 = Toman).
